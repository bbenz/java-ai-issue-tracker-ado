# Prompt: ADO Security Issue Resolution Tracker

## Goal

Build a **Python CLI tool** that automates the triage of security vulnerability issues tracked as Azure DevOps (ADO) work items. The tool parses vulnerability descriptions generated by security scanners (e.g., Snyk, Dependabot), checks whether each vulnerability has already been resolved by dependency updates in the latest stable releases of **Langchain4j** and **Spring AI**, produces a human-reviewable markdown report, and — after approval — bulk-updates ADO work items as resolved.

---

## Context

- **ADO Organization**: `YOUR_ADO_ORG`
- **ADO Project**: `YOUR_ADO_PROJECT`
- **ADO Query ID**: `YOUR_SAVED_QUERY_GUID`
- **Work Item Type**: Issue (state = "To Do", priority = 1)
- **Issue Source**: Security scanner (e.g., Snyk) — descriptions follow a consistent format (see Parsing section)
- **Target Projects**:
  | Project | Version | Repo |
  |---------|---------|------|
  | Langchain4j | 1.12.2 | https://github.com/langchain4j/langchain4j |
  | Spring AI | 1.1.4 | https://github.com/spring-projects/spring-ai |

---

## Tool Architecture

### Two-Phase Workflow

#### Phase 1 — Analysis (offline, no ADO writes)

1. **Fetch work items** from the saved ADO query (query ID above). Filter to state = "To Do" and priority = 1.
2. **Parse each work item description** to extract vulnerability details (see Parsing section below).
3. **Determine the parent project** (Langchain4j or Spring AI) using the ADO work item's **Project** field:
   - `SpringAI` → **Spring AI**
   - `LangChain4j` → **Langchain4j**
   - If the field is missing or unrecognized → **Flag for manual review**
   - *Note: Do NOT infer the parent project from the vulnerability description — use the ADO field as the source of truth.*
4. **Check if the vulnerability is resolved** by comparing the vulnerable package version against the dependency tree of the target project's latest stable release:
   - Fetch the effective dependency tree for Langchain4j 1.12.2 and Spring AI 1.1.4 (see Dependency Resolution section).
   - If the vulnerable package is no longer present, or its version is >= the scanner's "fixed in" version → **Resolved**.
   - If the vulnerable package is still present at a version below the fix version → **Unresolved**.
   - If the issue cannot be mapped to a dependency (e.g., code-level vulnerability, configuration issue) → **Flag as non-dependency issue**.
5. **Classify each issue** into one of these categories:
   - `RESOLVED_BY_DEPENDENCY_UPDATE` — The dependency has been updated to a version at or above the fix version.
   - `RESOLVED_DEPENDENCY_REMOVED` — The vulnerable dependency is no longer in the dependency tree.
   - `UNRESOLVED_DEPENDENCY` — The vulnerable dependency is still present at a vulnerable version.
   - `NON_DEPENDENCY_ISSUE` — The issue is not related to a dependency version (flag for investigation).
   - `PARSE_ERROR` — Could not parse the vulnerability description (flag for manual review).
6. **Generate a markdown report** (`analysis-report.md`) with the following structure:

```markdown
# ADO Security Issue Resolution Report
Generated: {timestamp}
Query: {query_url}

## Summary
- Total work items analyzed: {count}
- Resolved (will be marked Done): {count}
- Unresolved (requires attention): {count}
- Non-dependency issues (flagged): {count}
- Parse errors (manual review): {count}

## Priority 1

### Unresolved Issues
| ADO ID | Title | Severity | Vulnerable Package | Current Version | Fix Version | Category | Parent Project | Action |
|--------|-------|----------|-------------------|-----------------|-------------|----------|----------------|--------|
| 204458 | Denial of Service (DoS) | Medium | com.squareup.okio:okio-jvm | 3.0.0 | 3.4.0 | UNRESOLVED_DEPENDENCY | Spring AI | Needs fix |

### Non-Dependency Issues (Flagged)
| ADO ID | Title | Severity | Description Excerpt | Category | Action |
|--------|-------|----------|-------------------|----------|--------|

### Resolved Issues (Will Be Marked Done)
| ADO ID | Title | Severity | Vulnerable Package | Fixed Version | Category | Parent Project | Resolution Detail |
|--------|-------|----------|-------------------|---------------|----------|----------------|-------------------|
| 204337 | ... | Low | org.jetbrains.kotlin:kotlin-stdlib | 2.1.0 | RESOLVED_BY_DEPENDENCY_UPDATE | Spring AI | Updated to 2.1.0 in Spring AI 1.1.4 |

### Parse Errors (Manual Review)
| ADO ID | Title | Raw Description (first 200 chars) |
|--------|-------|------------------------------------|
```

*Note: If additional priority levels exist, repeat the grouping structure for each priority level, ordered by priority (1 first).*

#### Phase 2 — Live Mode (ADO writes, after human review)

1. **Read the analysis report** and present a summary to the user.
2. **Prompt for confirmation**: "Update {N} work items as resolved? [y/N/select]"
   - `y` — Update all resolved items.
   - `N` — Cancel, no changes.
   - `select` — Open an interactive selection (checkboxes or comma-separated IDs) to choose which items to update.
3. **For each item marked for update**:
   - Change the work item **State** from "To Do" to "Done".
   - Add a **Discussion comment** explaining the resolution:
     ```
     Automatically resolved by ADO Security Issue Tracker.
     Category: RESOLVED_BY_DEPENDENCY_UPDATE
     Detail: {vulnerable_package} updated from {old_version} to {new_version} in {parent_project} {project_version}.
     Advisory URL: {advisory_url}
     Verified: {timestamp}
     ```
4. **Generate an update log** (`update-log.md`) documenting all changes made.

## Vulnerability Description Parsing

The scanner-generated descriptions (e.g., from Snyk) follow this general pattern (may contain multiple vulnerabilities per work item):

```
{Vulnerability Title} [{Severity} Severity][{snyk_url}] in {package_group}:{package_name}@{vulnerable_version}
introduced by {root_dependency}@{root_version} > {intermediate}@{version} > ... > {vulnerable_package}@{version}
This issue was fixed in versions: {fix_version}
```

**Extract for each vulnerability:**
- `title` — e.g., "Denial of Service (DoS)"
- `severity` — e.g., "Medium"
- `snyk_url` — e.g., "https://security.snyk.io/vuln/SNYK-JAVA-COMSQUAREUPOKIO-5820002"
- `vulnerable_package` — groupId:artifactId, e.g., "com.squareup.okio:okio-jvm"
- `vulnerable_version` — e.g., "3.0.0"
- `dependency_chain` — the full transitive path
- `fix_version` — e.g., "3.4.0"
- `root_dependency` — the top-level dependency that introduces the vulnerability

**Handle edge cases:**
- Multiple vulnerabilities in a single work item description (split and check each)
- Missing or malformed fields → classify as `PARSE_ERROR`
- Severity values: Critical, High, Medium, Low

## Dependency Resolution

To check whether a vulnerability is resolved, the tool needs the effective dependency tree of each target project at its specified release version.

**Approach (in order of preference):**

1. **GitHub API + POM inspection (Primary)**: Fetch `pom.xml` files from the release tag via GitHub API. Parse `<dependencyManagement>`, `<dependencies>`, and `<properties>` sections across the multi-module project to determine declared and managed dependency versions. This is the primary approach — no local Maven installation required.

2. **Maven Central REST API (Fallback)**: If the GitHub POM inspection is insufficient to resolve a transitive dependency version (e.g., version inherited from a parent POM not in the repo), query the Maven Central Search API (`https://search.maven.org/solrsearch/select`) to look up artifact versions and verify whether a fixed version exists.

**Do NOT require a local Maven installation.** The tool should work entirely via HTTP APIs.

Cache the resolved dependency data so it only needs to be fetched once per run.

**Important**: Both Langchain4j and Spring AI are multi-module Maven projects. The tool should:
- Identify which module(s) contain the vulnerable dependency
- Check the BOM / `<dependencyManagement>` section for managed versions
- Resolve version properties (e.g., `${okio.version}`) by parsing `<properties>` blocks
- Walk parent POMs within the same GitHub repo to resolve inherited versions

**GitHub release tags to fetch POMs from:**
- Langchain4j 1.12.2: `https://github.com/langchain4j/langchain4j/tree/{release_tag}`
- Spring AI 1.1.4: `https://github.com/spring-projects/spring-ai/tree/{release_tag}`

*The tool should auto-detect the release tag format (e.g., `v1.12.2`, `1.12.2`, `langchain4j-1.12.2`) by listing tags via the GitHub API.*

## ADO Integration

### Authentication
- **Primary**: Azure CLI (`az login`) with Entra ID — use `az boards` commands or REST API with the bearer token from `az account get-access-token --resource 499b84ac-1321-427f-aa17-267ca6975798`
- **Fallback**: Personal Access Token (PAT) via environment variable `ADO_PAT`

### Read Work Items
```
az boards query --id YOUR_SAVED_QUERY_GUID --org https://dev.azure.com/YOUR_ADO_ORG --project YOUR_ADO_PROJECT
```
Then fetch full details for each work item ID returned.

### Update Work Items
```
az boards work-item update --id {id} --state Done --org https://dev.azure.com/YOUR_ADO_ORG --project YOUR_ADO_PROJECT --discussion "Resolution comment here"
```

## CLI Interface

```bash
# Phase 1: Analysis only (safe, no ADO writes)
python ado_issue_tracker.py analyze \
  --org YOUR_ADO_ORG \
  --project YOUR_ADO_PROJECT \
  --query-id YOUR_SAVED_QUERY_GUID \
  --output analysis-report.md

# Phase 2: Apply updates after reviewing the report
python ado_issue_tracker.py update \
  --report analysis-report.md \
  --org YOUR_ADO_ORG \
  --project YOUR_ADO_PROJECT \
  --dry-run          # Preview changes without applying
  
python ado_issue_tracker.py update \
  --report analysis-report.md \
  --org YOUR_ADO_ORG \
  --project YOUR_ADO_PROJECT \
  --confirm          # Actually apply changes
```

### Configuration
All parameters should be configurable via:
1. CLI arguments (highest priority)
2. Environment variables (`ADO_ORG`, `ADO_PROJECT`, `ADO_QUERY_ID`, `ADO_PAT`)
3. A `.env` file in the project root

### Target Project Versions
The tool should accept target project versions as arguments or from a config file:
```bash
--targets "langchain4j:1.12.2,spring-ai:1.1.4"
```

## Error Handling & Safety

- **Never write to ADO without explicit user confirmation.**
- Log all operations to a timestamped log file.
- If a work item update fails, log the error and continue with remaining items (don't abort the batch).
- Rate-limit ADO API calls to avoid throttling.
- Validate that the ADO query returns only "Issue" type work items with state "To Do" and priority 1 before proceeding.

## Design Decisions

1. **Multi-vulnerability work items**: If a work item contains multiple vulnerability issues and some are resolved but others aren't, the work item remains "To Do" (conservative). The report notes the mixed status, listing which vulnerabilities are resolved and which aren't.

2. **Parent project mapping**: Uses the ADO work item's **Project** field (`SpringAI` or `LangChain4j`). If the field is missing or unrecognized, classify as `PARSE_ERROR` and include in the manual review section.

3. **Dependency tree caching**: Persist fetched POM data to disk in `dependency-cache/` with a configurable TTL (default: 7 days). Force refresh with `--no-cache`.

4. **No local Maven required**: All dependency resolution is done via GitHub API (primary) and Maven Central REST API (fallback).
